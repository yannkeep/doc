Présentation générale

#B!Mi Plaidoyer Citoyen est une application web monopage (Single-Page Application) conçue comme un poste de travail numérique complet pour la construction de stratégies de plaidoyer citoyen. Elle est distribuée sous forme d'un unique fichier HTML auto-contenu, sans nécessiter de serveur ni de connexion réseau après chargement initial.

L'application implémente la méthodologie Voir-Juger-Agir (VJA), issue de la tradition de l'éducation populaire et de l'action communautaire, en la décomposant en 15 outils structurés répartis sur trois phases d'analyse et d'action.

Public cible

Militants, associations, collectifs citoyens, mouvements d'éducation permanente, facilitateurs de démocratie participative — toute personne ou organisation cherchant à formaliser une stratégie de changement social.

URL de référence

L'application est hébergée à l'adresse canonique : https://yannkeep.github.io/plaidoyer/app/index.html

Architecture technique

Fichier unique, zéro dépendance backend

L'intégralité de l'application — HTML, CSS, JavaScript, logique métier, modèle de données — est contenue dans un seul fichier .html. Aucune requête API externe n'est effectuée en dehors de la police de caractères Google Fonts et des bibliothèques CDN chargées au démarrage.

Structure du DOM

L'application est organisée autour d'un layout en deux colonnes :

  • Sidebar (#sidebar) — navigation entre les 15 outils, indicateur de statut, bouton raccourcis clavier
  • Zone principale (#main-content) — affichage conditionnel des sections (.page) via showPage()
  • Header (#header) — titre du projet actif, actions globales (nouveau, dupliquer, supprimer)
  • Système de toasts — notifications non-bloquantes en haut de l'écran

Routage interne

Chaque outil correspond à une <section class="page" id="p-[nom]">. La navigation est gérée par showPage(pageId) qui masque/affiche les sections en manipulant la classe active. Aucun hash URL ou History API n'est utilisé.

Persistance des données

L'application gère plusieurs projets en parallèle grâce à une couche de persistance locale à deux niveaux avec fallback automatique.

IndexedDB (prioritaire)

Stockage structuré côté navigateur, sans limite de taille pratique. Chaque projet est stocké comme un enregistrement dans le store projects. Les opérations sont asynchrones (async/await).

localStorage (fallback)

Utilisé automatiquement si IndexedDB est indisponible (navigateurs anciens, mode privé strict). Tous les projets sont sérialisés dans une clé unique bimi_projects sous forme de tableau JSON.

API de persistance interne

FonctionDescription
openDB()Ouvre la base IndexedDB (version 1), crée le store si absent
dbGetAll()Retourne tous les projets triés par date de modification
dbGet(id)Récupère un projet par son id unique
dbPut(project)Crée ou met à jour un projet (upsert)
dbDelete(id)Supprime définitivement un projet

Sauvegarde automatique

Chaque modification d'un champ (input, textarea) déclenche scheduleSave(), qui met en place un debounce de 1500 ms avant la sauvegarde effective. Ceci évite les écritures trop fréquentes sans risquer de perdre des données.

Note : Les données sont exclusivement stockées localement dans le navigateur. Aucune information ne transite vers un serveur externe. La confidentialité des projets est donc totale.

PWA & mode hors-ligne

L'application implémente les spécifications PWA (Progressive Web App) pour permettre une installation native sur mobile et desktop, ainsi qu'un fonctionnement hors-ligne complet.

Gestion du statut réseau

Des listeners online / offline affichent des toasts de notification et appliquent la classe offline sur le body pour adapter l'interface visuellement. En mode hors-ligne, toutes les fonctionnalités de saisie et d'export restent disponibles.

Mise à jour

Lorsqu'une nouvelle version du Service Worker est détectée (statechange → installed), une notification de mise à jour est proposée via showUpdateNotification().

Cadre méthodologique Voir-Juger-Agir

La méthodologie Voir-Juger-Agir est une démarche d'action collective née dans la tradition de l'éducation populaire catholique (JOC, années 1930, Joseph Cardijn). Elle structure le processus de changement social en trois étapes séquentielles et itératives :

VOIR — Observer et analyser la réalité concrète, identifier les acteurs et les rapports de pouvoir, recueillir les faits sans jugement préalable.
JUGER — Interpréter les faits à la lumière de valeurs et d'outils d'analyse, élaborer une lecture critique et stratégique de la situation.
AGIR — Définir et mettre en œuvre des actions concrètes, formuler des objectifs mesurables, construire des alliances et communiquer.

Les 15 outils de l'application sont répartis équitablement entre ces trois phases : 3 outils VOIR + 6 outils JUGER + 6 outils AGIR.

👁️
VOIR — Analyse
3 outils · Observer la réalité, identifier les acteurs et les rapports de pouvoir
🎯 Vision (Domino du changement)
Clarification stratégique fondamentale. Trois champs : Vision (le changement concret souhaité), Obstacles (freins identifiés), Ressources (forces mobilisables). C'est l'outil de cadrage initial du projet.
👤 Profil citoyen
Autodiagnostic de l'engagement. Quatre champs : Motivations, Compétences, Disponibilité, Limites. Permet de calibrer l'engagement sur la durée et d'identifier les besoins de formation.
🌸 Fleur du pouvoir
Analyse intersectionnelle des rapports de pouvoir. Trois axes : Identités des personnes concernées, Privilèges du collectif porteur, Oppressions en jeu. Outil d'ancrage dans une perspective de justice sociale.
⚖️
JUGER — Stratégie
6 outils · Analyse critique, modélisation du problème et construction stratégique
👥 Cartographie des acteurs
Identification et positionnement de toutes les parties prenantes. Matrice Pouvoir/Intérêt, visualisation en réseau interactif (vis.js), gestion des liens entre acteurs. Quatre quadrants stratégiques : Gérer de près / Garder satisfait / Garder informé / Surveiller.
🌳 Arbre à problèmes
Modélisation visuelle du problème central (tronc), ses causes profondes (racines) et ses conséquences (branches). Permet d'identifier les points de levier prioritaires pour l'action.
❓ Les 5 Pourquoi
Technique d'analyse causale par itération. Cinq champs séquentiels permettant de remonter aux causes racines structurelles au-delà des symptômes apparents.
📐 Analyse SWOT
Grille classique en quatre quadrants : Forces internes / Faiblesses internes / Opportunités externes / Menaces externes. Synthèse stratégique du contexte d'action.
🌍 Analyse PESTEL
Balayage macro-environnemental en six dimensions : Politique, Économique, Social, Technologique, Environnemental, Légal. Indispensable pour contextualiser le plaidoyer dans son environnement systémique.
🔄 Théorie du changement
Explicitation du chemin de changement : situation actuelle → situation souhaitée, via mécanismes d'action et hypothèses sous-jacentes. Quatre champs : Situation actuelle, Situation souhaitée, Mécanismes, Hypothèses.
✊
AGIR — Action
6 outils · Planification, communication et suivi de l'impact
⚡ Avec / Sans / Contre
Trois postures stratégiques vis-à-vis du pouvoir : Avec (collaboration, lobbying institutionnel), Sans (alternatives parallèles, projets pilotes), Contre (opposition, mobilisation, recours juridiques). Outil de décision tactique.
🎯 Cibles & Alliés
Identification précise des trois cercles : Cibles principales (décideurs ayant le pouvoir de changement), Cibles secondaires (influenceurs), Alliés (partenaires stratégiques). Base de la stratégie d'influence.
📋 Objectifs SMART
Gestionnaire complet d'objectifs SMART (Spécifique, Mesurable, Atteignable, Réaliste, Temporel). Avec indicateur de mesure, échéance, statut kanban (À faire / En cours / Fait / Abandonné), vues liste / kanban / timeline / Gantt.
💬 Message clé
Construction du message de plaidoyer selon la structure classique : Accroche → Problème → Pourquoi c'est important → À qui on s'adresse → Action demandée. Prévisualisation temps réel et copie presse-papiers.
📈 Suivi-évaluation
Cadre d'évaluation de l'impact : Indicateurs de succès (quantitatifs et qualitatifs), Méthodes de collecte, Calendrier d'évaluation, Leçons apprises. Ancrage dans une culture de redevabilité.
📓 Journal de bord
Chronique datée du parcours militant. Entrées horodatées, texte libre. Mémoire collective du projet, précieuse pour la transmission et l'évaluation rétrospective.

Cartographie des acteurs — détail technique

La cartographie est l'outil le plus complexe techniquement. Elle combine une gestion de liste tabulaire avec une visualisation en graphe de réseau interactif via la bibliothèque vis.js Network.

Modèle d'un acteur

{
  id:       number,      // Identifiant unique auto-incrémenté
  name:     string,      // Nom de l'acteur
  role:     string,      // Rôle / fonction
  position: string,      // "ally" | "opponent" | "neutral" | "target"
  power:    number       // Score de pouvoir 1-5 (matrice Pouvoir/Intérêt)
}

Visualisation réseau

Le graphe vis.js est initialisé par initNetworkGraph() et mis à jour par updateNetworkGraph() à chaque modification. Les liens entre acteurs sont gérés séparément (project.links). Un bouton d'export en image PNG est disponible directement dans l'interface.

Vues disponibles

L'outil propose deux vues commutables : Liste (tableau enrichi avec drag-and-drop pour réorganiser) et Réseau (graphe interactif force-directed). La fonction switchActorView(view) gère la transition.

Codage couleur des échéances

La fonction getDeadlineClass(deadline) applique une classe CSS selon l'urgence : overdue (dépassé) · urgent (< 7 jours) · soon (< 14 jours) · défaut (neutre).

Journal de bord

Modèle d'une entrée

{
  id:      number,
  date:    string,   // "YYYY-MM-DD"
  content: string    // Texte libre de l'entrée
}

Les entrées sont affichées en ordre chronologique inverse (la plus récente en premier). La date est automatiquement initialisée à aujourd'hui lors de la création d'une nouvelle entrée.

Formats d'export disponibles

📋 JSON 📝 Markdown 🌐 HTML 📊 CSV 📄 PDF stylisé 📘 Word .docx

DOCX generateDOCX() Document Word avec styles professionnels, structuré par phases VJA. Utilise la bibliothèque docx.js.

Import JSON

La fonction importJSON(file) accepte :

  • Les exports natifs #B!Mi (format { project: {...}, ... })
  • Les datasets compatibles (ancien format plat)
  • Les fichiers glissés-déposés sur la zone de drop dédiée
Format non reconnu : Si le fichier JSON importé ne correspond ni au format natif ni à un dataset compatible, l'application lève une erreur explicite : « Format de fichier non reconnu. Utilisez un export JSON de #B!Mi ou un dataset compatible. »

  desc:       string,       // Description courte

  updatedAt:  string,       // ISO date (mis à jour à chaque save)

  // ── VOIR ──

    vision:     string,     // Le changement concret souhaité

  profil: {

  // ── JUGER ──

Modèles préconfigurés

L'application propose deux modèles de projet entièrement préconfigurés, accessibles depuis le tableau de bord lors de la création d'un projet. Ils servent à la fois de démonstration des capacités de l'outil et de bases de travail réelles.

🛑 STOP Arizona — Défense du Modèle Social Belge

Projet de plaidoyer contre la limitation des allocations de chômage à 24 mois instaurée par la coalition Arizona. Le modèle intègre une vision DOMINO complète, une cartographie d'acteurs (De Wever, Vandenbroucke, FGTB, Cour Constitutionnelle), des objectifs SMART avec échéances 2026, et tous les outils d'analyse pré-remplis avec les données du contexte belge.

💡 Économie Contributive Provisionnée (ECP)

Modèle le plus complet — l'intégralité des 15 outils est pré-remplie avec les données du projet ECP visant à obtenir un ruling fiscal favorable du SDA (Service des Décisions Anticipées). Inclut : analyse PESTEL du contexte Arizona, théorie du changement détaillée, cartographie de 10 acteurs clés, 5 objectifs SMART chiffrés, message de plaidoyer complet, cadre d'évaluation avec indicateurs qualitatifs et quantitatifs.

Ce modèle constitue également une démonstration de l'utilisation de l'application pour un projet de politique sociale complexe impliquant des enjeux juridiques, institutionnels et militants.
